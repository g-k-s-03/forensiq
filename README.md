# ForensIQ -- Digital Forensics Evidence Analyzer

ForensIQ is a Python CLI tool that performs automated digital forensics analysis on an evidence directory. It extracts file metadata and EXIF data, computes SHA-256 hashes for tamper detection, recovers deleted browser history from SQLite freelist pages and WAL files, builds a unified chronological timeline across all artifact sources, and generates a court-format PDF report pre-structured for submission under Section 65B of the Indian Evidence Act, 1872.

---

## Legal Disclaimer

This tool generates forensic worksheets for examiner review. All findings are observational and rule-based. ForensIQ does not perform inference or draw legal conclusions. Reports generated by this tool **require physical signature and Section 65B certification by a qualified forensic examiner** before submission to any court or authority. The examining investigator is solely responsible for the accuracy of findings and their interpretation in a legal context.

## Features

### Phase 1 -- File Metadata Extraction
- Recursively scans all files in an evidence directory
- Extracts name, extension, size, and created/modified/accessed timestamps
- Detects timestamp anomalies: `mtime < ctime` (possible backdating), `atime < mtime`
- Identifies hidden files (dot-prefixed names)
- Extracts EXIF metadata from `.jpg`, `.jpeg`, `.tiff`, `.png` files (camera make/model, GPS, date taken)

### Phase 2 -- Hash Checker & Tamper Detection
- SHA-256 hashes every file in 64 KB chunks (efficient for large evidence)
- Saves a hash manifest (`output/{case_id}_hashes.json`) on every run
- Compares against a saved baseline to detect `HASH_MISMATCH`, `NEW_FILE`, `DELETED_FILE`
- Flags `TIMESTAMP_ANOMALY` and `HIDDEN_FILE` regardless of baseline
- Color-coded severity table: HIGH=red, MEDIUM=yellow, LOW=cyan

### Phase 3 -- Hash Integrity Engine (Deep Analysis)
- Every flag cites a specific forensic rule (R-01 through R-10)
- **Entropy analysis**: Shannon entropy on plaintext file types; `HIGH_ENTROPY` if > 7.2
- **Magic bytes detection**: reads first 16 bytes against 8 known signatures (JPEG, PNG, PDF, ZIP, OLE, RTF, PE, ELF); raises `DISGUISED_FILE` or `SUSPICIOUS_EXECUTABLE`
- **EXIF strip detection**: `EXIF_STRIPPED` for JPEG files with no EXIF tags
- **Anti-forensic detection**: `ANTI_FORENSIC` when hash changed but timestamps appear unmodified

### Phase 4 -- Browser Forensics & Deleted History Recovery
- Live extraction from Chrome/Edge (`History`, `Cookies`) and Firefox (`places.sqlite`, `cookies.sqlite`)
- Always works on file copies to avoid locking live browser databases
- **WAL recovery**: copies `-wal` companion file so SQLite auto-merges uncommitted records
- **Freelist scanning**: regex-based URL recovery from SQLite unallocated pages; deduplicated against live records
- **DNS cache snapshot**: `ipconfig /displaydns` (Windows) to extract resolved hostnames
- **Prefetch evidence**: lists Chrome/Firefox/Edge `.pf` files with last-execution timestamps

### Phase 5 -- Forensic Timeline Builder
- Merges all artifact sources into a single chronological event list
- Event types: `FILE_CREATED`, `FILE_MODIFIED`, `BROWSER_VISIT`, `BROWSER_DOWNLOAD`, `BROWSER_COOKIE`, `DNS_LOOKUP`, `BROWSER_EXECUTED`
- Confidence levels: `CONFIRMED` (live indexed), `INFERRED` (derived), `RECOVERED` (deleted/WAL)
- Unknown-timestamp events sort to the end
- Exports `output/{case_id}_timeline.json`
- Auto-generated Forensic Narrative with Section 65B disclaimer

### Phase 6 -- Court-Admissible PDF Report
- ReportLab A4 PDF with headers, footers, and "Page X of Y"
- 9 sections: Cover, Section 65B Certificate, Chain of Custody, Case Summary, File Inventory & Hash Manifest, Tamper Detection Findings, Browser Forensic Findings, Forensic Timeline, Examiner Declaration
- Color-coded severity and confidence in tables; alternating row shading
- Examiner Declaration with signature block and IMPORTANT note box
- Zero AI-generated conclusions -- all findings are observational

### Phase 7 -- Integration & Polish
- `--no-report` flag to skip PDF on quick analysis runs
- `--json` flag to export full structured analysis as `output/{case_id}_full_analysis.json`
- Graceful error handling: Browser Forensics, Timeline, and PDF sections each wrapped in try/except; pipeline continues on partial failure with `[WARN]` messages

## Usage

### `analyze` -- Scan and detect

```bash
python main.py analyze <evidence_dir> --case-id <ID> --investigator "<Name>" [options]
```

| Option             | Description                                     | Default          |
|--------------------|-------------------------------------------------|------------------|
| `evidence_dir`     | Path to evidence directory                      | *(required)*     |
| `--case-id`        | Case identifier                                 | *(required)*     |
| `--investigator`   | Investigator name                               | *(required)*     |
| `--device`         | Device description                              | `Unknown Device` |
| `--output`         | Output directory                                | `output/`        |
| `--baseline`       | Path to baseline hashes JSON for comparison     | *(none)*         |
| `--json/--no-json` | Export full structured analysis as JSON         | `--no-json`      |
| `--report/--no-report` | Generate PDF report                        | `--report`       |

### `save-baseline` -- Snapshot current hashes

```bash
python main.py save-baseline <evidence_dir> [--output <file>]
```

| Option         | Description                  | Default                |
|----------------|------------------------------|------------------------|
| `evidence_dir` | Path to evidence directory   | *(required)*           |
| `--output`     | Output JSON file path        | `baseline_hashes.json` |

### Tamper Detection Workflow

```bash
# 1. Snapshot clean state
python main.py save-baseline ./sample_evidence --output baseline.json

# 2. (time passes -- suspect may have modified files)

# 3. Re-analyze against baseline
python main.py analyze ./sample_evidence \
    --case-id CASE-002 \
    --investigator "J. Smith" \
    --baseline baseline.json
```

### JSON Export Example

```bash
python main.py analyze ./sample_evidence \
    --case-id CASE-003 \
    --investigator "Gaurav Kumar" \
    --device "Suspect Laptop - Dell Inspiron" \
    --baseline baseline.json \
    --json \
    --no-report
```

Produces `output/CASE-003_full_analysis.json` with case metadata, summary counts, all tamper flags, browser history, downloads, and timeline (artifact payloads excluded).

## Forensic Rule Set

| Rule  | Flag                   | Severity | Trigger                                                   |
|-------|------------------------|----------|-----------------------------------------------------------|
| R-01  | `HASH_MISMATCH`        | HIGH     | SHA-256 digest differs from verified baseline             |
| R-02  | `NEW_FILE`             | MEDIUM   | File absent from baseline snapshot                        |
| R-03  | `DELETED_FILE`         | HIGH     | File present in baseline but missing from evidence        |
| R-04  | `TIMESTAMP_ANOMALY`    | MEDIUM   | mtime < ctime or atime < mtime                            |
| R-05  | `HIDDEN_FILE`          | LOW      | Dot-prefixed filename used for concealment                |
| R-06  | `HIGH_ENTROPY`         | MEDIUM   | Shannon entropy > 7.2 in a plaintext file type            |
| R-07  | `DISGUISED_FILE`       | HIGH     | File extension does not match internal magic bytes        |
| R-08  | `SUSPICIOUS_EXECUTABLE`| HIGH     | .exe / .dll found in evidence collection                  |
| R-09  | `EXIF_STRIPPED`        | MEDIUM   | JPEG image lacks EXIF metadata                            |
| R-10  | `ANTI_FORENSIC`        | HIGH     | Content changed (hash mismatch) but timestamps unmodified |

## Technical Notes

- **Browser DB copies**: always copied to a private `tempfile.mkdtemp()` before opening to avoid locking live browser databases; `-wal` and `-shm` companions are copied alongside so SQLite auto-merges the WAL on connection open
- **Freelist scanning**: regex `rb"https?://[\x21-\x7e]{4,255}"` applied to raw DB bytes; results deduplicated against live URLs using prefix-match (handles SQLite B-tree page layout where URL and title are stored adjacently)
- **Chrome timestamps**: microseconds since 1601-01-01 (Windows FILETIME epoch)
- **Firefox timestamps**: microseconds since Unix epoch
- **Entropy analysis**: Shannon entropy over 256-bucket byte frequency distribution; threshold 7.2 for plaintext extensions (`.txt`, `.log`, `.csv`, `.xml`, `.html`, `.js`, `.py`)
- **Magic bytes**: first 16 bytes matched against JPEG (FFD8FF), PNG (89504E47), PDF (%PDF), ZIP/Office (504B), OLE (D0CF), RTF ({\\rtf), PE (4D5A), ELF (7F454C46)
- **PDF "Page X of Y"**: two-pass numbered canvas -- page states captured on first pass, footer stamped with total during `save()`
- **All tamper flags cite R-01..R-10**: deterministic, rule-based, no statistical inference

## Quick Start (Production — Full Web App)

Builds the React frontend and serves everything from a single FastAPI server:

```powershell
cd forensiq
pip install -r requirements.txt
python start.py          # builds frontend then starts API at http://localhost:8000
```

Open `http://localhost:8000` in your browser.

## Quick Start (Development — Hot Reload)

Run backend and frontend separately for faster iteration:

```powershell
# Terminal 1 — API (with hot reload)
cd forensiq
pip install -r requirements.txt
python run_api.py        # http://localhost:8000

# Terminal 2 — Frontend (with hot module replacement)
cd forensiq/frontend
npm install
npm run dev              # http://localhost:5173
```

## Running the Web API

API available at `http://localhost:8000` (production) or `http://localhost:8000` / `http://localhost:5173` (dev)

| Endpoint | Method | Description |
|---|---|---|
| `/api/health` | GET | Health check |
| `/api/analyze` | POST | Start a background analysis job |
| `/api/analyze/{job_id}` | GET | Poll job status (0-100% progress) |
| `/api/analyze/{job_id}/stream` | GET | Server-Sent Events live progress stream |
| `/api/analyze/{job_id}/results` | GET | Retrieve full results when complete |
| `/api/baseline` | POST | Save a hash baseline |
| `/api/reports/list` | GET | List all case IDs with output files |
| `/api/reports/{case_id}/pdf` | GET | Download PDF report |
| `/api/reports/{case_id}/json` | GET | Download full JSON export |
| `/api/reports/{case_id}/timeline` | GET | Download timeline JSON |
| `/api/reports/{case_id}/hashes` | GET | Download hash manifest |

### Example: run a full analysis via API

```bash
# Start job
curl -s -X POST http://localhost:8000/api/analyze \
  -H "Content-Type: application/json" \
  -d '{"evidence_dir":"./sample_evidence","case_id":"API-001","investigator":"Gaurav Kumar","generate_pdf":true,"export_json":true}'

# Poll status
curl -s http://localhost:8000/api/analyze/<job_id>

# Get results
curl -s http://localhost:8000/api/analyze/<job_id>/results

# List all reports
curl -s http://localhost:8000/api/reports/list
```
